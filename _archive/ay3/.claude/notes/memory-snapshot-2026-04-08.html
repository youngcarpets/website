Reference ref_phone_dev_workflow.md

How to view the live Vite dev server on a phone from anywhere — Cloudflare quick tunnel + cloudflared standalone binary, no admin/UAC required.

Why not LAN IP (http://10.0.0.x:5173)?

  • Windows Firewall blocks inbound on dev ports by default (needs UAC to fix)
  • DHCP rotates both phone and laptop IPs — IP changes break the URL constantly
  • Only works on same Wi-Fi — useless on cellular or away from home

The setup (one-time)

  2. vite.config.ts has server.allowedHosts permitting tunnel domains:

  allowedHosts: ['.trycloudflare.com', '.loca.lt', '.ngrok-free.app']
}
    Without this Vite returns "Blocked request. This host is not allowed." Committed in v1.20.7 on 2026-04-07.

The workflow (every session)

  1. Start Vite with --host (binds to all interfaces, required for tunnel to reach it):

  2. Start the quick tunnel pointing at port 5173 (run in background):
    "c:/Users/alany/.claude/bin/cloudflared.exe" tunnel --url http://localhost:5173 --no-autoupdate
  3. Read the tunnel output for the random https://*.trycloudflare.com URL. Hand it to the user — append the route they want.

Caveats

  • Quick-tunnel URLs are ephemeral — die when cloudflared stops or laptop sleeps. New URL every restart.
  • Hot reload works over the tunnel (Vite HMR via WebSocket through Cloudflare).
  • The phone needs no special config — just paste the URL.

Multi-Agent Async Flow

Feedback High Priority LOVE'd feedback_agent_flow.md

Dispatch background agents in parallel; user keeps adding todos while they work; agents recursively dispatch sub-agents for big tasks; main thread compiles findings.

User said on 2026-04-07: "i LOVE this flow. where you send off agent and i can keep adding to todo list without messing things up. LOVE IT." This is now the standard operating mode for any non-trivial work in this project.

The flow

  1. User gives a directive that touches multiple domains OR a task that needs cross-checking (build → review → fix).
  2. Main thread immediately:
    • Updates the TodoWrite list to capture the directive + every sub-step
    • Dispatches parallel background agents — one per domain — in the SAME message via multiple Agent tool calls
    • Each agent gets the FULL context it needs and explicit instructions to dispatch sub-agents recursively if its task is large
    • Returns a short status to the user: "X agents dispatched, standing by"
  3. While agents run:
    • User keeps adding todos / new requests / refinements — these stack into the TodoWrite list
    • Main thread NEVER blocks waiting on agents — it can do small mechanical work IF the user asks
    • Main thread silently catches <task-notification> messages as agents complete
  4. As each agent reports back:
    • Findings get noted in the running todo
    • If any agent's output requires user input before the next phase, surface ONE question (slow-discovery rule)
  5. When all agents in a phase have landed:
    • Main thread compiles their findings into a coherent implementation plan
    • Implementation runs in chunks per the chunking rule
  6. After implementation:
    • Dispatch parallel re-review agents (error check, design re-review, accessibility, etc.) — same async pattern
    • Fix anything they catch
    • Ship the version bump

Sub-agent recursion rule

Per user 2026-04-07: "when agents are called to do things.. they should also call other appropriate sub agents if the task is large."

When dispatching an agent, tell it explicitly that it can spawn its own background sub-agents if its task decomposes into independent work. Give it permission to fan out. The sub-agents return to the parent agent, the parent compiles, the parent returns to me. This is a tree of agents, not a flat list.

Why this matters

  • The user can keep thinking and adding to the queue without the conversation feeling stuck
  • Agents don't block each other (parallel) and don't block the user
  • Big work decomposes naturally into a tree instead of a serial chain
  • Verification (re-review agents) is the same primitive as research (planning agents)

When this flow applies

Use it for: anything that touches 3+ files or 2+ domains, anything that needs research before implementation, anything that needs verification after implementation, anything where the user has said "research" / "look into" / "figure out" / "make this work".

Do NOT use it for: a single-file edit the user explicitly directed, a single mechanical task ("commit and push"), trivial fixes that take 30 seconds.

Master Todo + Interrupt-Aware Ordering

Feedback High Priority LOVE'd feedback_master_todo.md

TodoWrite is the master queue; every interrupt gets captured and slotted by dependency order, not append order; user owns priorities, Claude owns the dependency graph.

User said on 2026-04-07: "it's like you have a master todo list.. but i keep interrupting with other tasks that you add for future and put them in the correct logical flow of tasks if that makes sense."

1. TodoWrite IS the master list

  • Every directive — current work, future work, deferred, blocked, parked questions, re-review passes — lives in TodoWrite.
  • No mental tracking. If it's not in the list, it doesn't exist.
  • Every interrupt becomes an entry immediately.
  • The list mirrors to .claude/RELAY.md at every save / handoff.

2. Logical ordering, not chronological

When a new task arrives, BEFORE adding it to the list, do a 5-second dependency check:

  • Blocks on? Does this task need something earlier in the queue to ship first? → slot it after that item.
  • Batches with? Does it touch the same files / tokens / concepts as another queued item? → slot it adjacent so they ship in one chunk.
  • Unblocks? Does it unblock anything already in the queue? → slot it before that item.
  • Independent? No dependencies → slot by user intent.

Then announce where it landed in ONE sentence and return to the in-progress task.

3. User is the priority overrider

Claude owns the dependency graph. The user owns priorities. Shorthand:

  • "next" / "do next" → promote to position 1 after current finishes
  • "now" / "drop what you're doing" → pause current, switch immediately
  • "not now" / "later" → move to end or (DEFERRED BY USER)
  • "park it" → move to end with (PARKED — needs user input)

4. Status tags make state legible

Exactly ONE in_progress at a time. Use inline annotations:

  • (WAITING ON USER) — blocked on user decision/input
  • (DEFERRED BY USER) — explicitly parked
  • (BLOCKED ON: <task>) — blocked on another task in the queue
  • (AGENT IN FLIGHT: <id>) — background agent working on this
  • (IDEA — not confirmed) — user floated it but hasn't committed

5. In-flight agents don't pause the queue

If new tasks are independent of an in-flight agent's outcome → work on them. If they depend on it → queue them, stay idle or pick something else, don't preemptively start work an agent might invalidate.

6. Re-ordering is announced briefly (one sentence max)

One sentence, then back to work. No tables.

7. The list survives the conversation

At every natural pause — user says "save", every ~5–8 messages on long tracks, any handoff — persist the master list state into .claude/RELAY.md's 📌 LATEST pin.
